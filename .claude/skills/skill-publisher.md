# Skill: Skill 生成器与发布器 (Skill Generator & Publisher)

## 触发条件
当用户提到以下意图时激活：

**生成 Skill（核心新能力）：**
- "把这个打包成 skill"、"把刚才做的变成 skill"
- "生成一个 skill"、"create a skill from this"
- "我想让别人也能用这个流程"
- "把这个能力固化下来"

**打包发布（原有能力）：**
- "打包 skill"、"发布 skill"、"publish skill"
- "推送到 GitHub"、"push to GitHub"
- "把这个 skill 分享给别人"

**单独使用：**
- "检查一下有没有敏感信息"（仅安全检查）
- "帮我写个 README"（仅生成文档）

## 概述

两大核心能力：

1. **Skill 生成器**：用户在多轮对话中完成了某个任务后，回溯对话内容，提炼出可复用的 Skill 定义文件（`.claude/skills/*.md`）+ 配套代码
2. **Skill 发布器**：将 Skill 项目安全检查、打包、推送到 GitHub

可以单独使用任一能力，也可以串联：生成 → 打包 → 发布 一气呵成。

## 第一部分：Skill 生成器

### 工作原理

Claude 回溯当前对话，提取以下信息：

1. **用户完成了什么任务**：目标是什么、解决了什么问题
2. **工作流程**：分几步完成、每步做了什么
3. **使用了哪些工具**：Python 脚本、bash 命令、MCP 工具、WebSearch、WebFetch 等
4. **关键决策点**：哪里需要用户确认、哪里有分支逻辑
5. **输入和输出**：接收什么输入、产出什么结果
6. **代码产物**：对话中创建或修改了哪些代码文件

### 第 G1 步：分析对话，提炼流程

**回溯整个对话历史**，总结：

```
📋 任务分析
━━━━━━━━━━━
任务：[一句话描述用户完成了什么]
场景：[适用于什么场景/谁会用]
步骤：
  1. [步骤名] — [做了什么]
  2. [步骤名] — [做了什么]
  ...
工具/脚本：
  - [文件路径] — [用途]
  - [工具名] — [用途]
用户交互点：
  - [哪里需要用户确认/输入]
```

**展示给用户确认**：
> "根据我们刚才的对话，我提炼出了以上流程。
> 这个理解准确吗？有没有需要调整的地方？"

### 第 G2 步：设计 Skill 架构

根据任务复杂度，决定 skill 结构：

**简单任务（≤3 步）**→ 生成 1 个 skill 文件
**中等任务（4-6 步）**→ 生成 1 个主 skill，步骤写在同一个文件里
**复杂任务（>6 步或有明显独立子流程）**→ 生成 1 个编排 skill + 多个子 skill

**向用户说明架构决策**：
> "这个任务比较复杂，我建议拆成 X 个 skill：
> - `main-skill.md`：主流程编排
> - `sub-skill-1.md`：负责 [子功能]
> - `sub-skill-2.md`：负责 [子功能]
>
> 你觉得这样合理吗？或者你希望合成一个？"

### 第 G3 步：生成 Skill 文件

按照以下标准模板生成 `.claude/skills/xxx.md`：

```markdown
# Skill: [中文名] ([English Name])

## 触发条件
当用户提到以下意图时激活：
- "[关键词1]"、"[关键词2]"
- "[关键词3]"、"[关键词4]"

## 概述
[一段话说明这个 skill 做什么、适用于什么场景]

## 前置输入
开始前必须已知：
1. [需要什么输入]
2. [需要什么配置]

## 工作流程

### 第 1 步：[步骤名]
[具体说明做什么、怎么做]

**交互节点**：[需要用户确认什么]

### 第 2 步：[步骤名]
...

## 错误处理
[常见错误和应对]

## 输出格式
[最终产出是什么]
```

**生成原则**：

1. **触发条件要自然**：用用户日常会说的话，中英文都覆盖
2. **流程写清楚但不写死**：描述"做什么"而非"怎么做的代码"
3. **保留交互节点**：关键步骤必须有用户确认
4. **不在 skill 文件里嵌入大段代码**：代码放在 `scripts/` 目录，skill 里只写调用方式
5. **通用化**：去掉对话中的特定案例细节（公司名、产品名等），用占位符或通用描述
6. **确保独立可用**：skill 既能被其他 skill 调用，也能直接被用户触发

### 第 G4 步：处理配套代码

检查对话中是否创建了 Python 脚本或其他代码文件：

**如果有代码产物**：
1. 确认这些文件已保存在项目中（`scripts/` 目录）
2. 检查代码中是否有硬编码的个人信息（邮箱、路径、密码等）
3. 确保代码是通用的，去除特定业务逻辑
4. 如果有依赖，生成或更新 `requirements.txt`

**如果没有代码产物**：
- 仅生成 `.md` skill 文件即可
- 有些 skill 纯靠 Claude 的对话能力完成，不需要代码

### 第 G5 步：用户审核

将生成的 skill 文件完整展示给用户：

> "以下是生成的 Skill 文件，请审核：
>
> [完整展示 .md 内容]
>
> 你可以：
> 1. ✅ 确认，保存到 `.claude/skills/`
> 2. ✏️ 告诉我修改意见，我来调整
> 3. 🔄 重新生成（如果方向不对）"

用户确认后，保存到 `.claude/skills/` 目录。

### 第 G6 步：验证 Skill

保存后进行基本验证：
1. 确认文件已写入 `.claude/skills/` 目录
2. 如果有配套脚本，确认 `scripts/` 目录中的文件可用
3. 给出测试建议：

> "✅ Skill 已保存！你可以在新对话中试试说：
> - \"[触发关键词1]\"
> - \"[触发关键词2]\"
>
> 要不要现在继续打包发布到 GitHub？"

## 第二部分：Skill 发布器

### 第 P1 步：扫描项目结构

自动识别需要打包的文件：

```
需要打包：
✅ .claude/skills/*.md — Skill 定义文件
✅ scripts/**/*.py — 配套 Python 模块
✅ scripts/**/requirements.txt — Python 依赖
✅ README.md — 如果存在
✅ install.sh — 如果存在
✅ .gitignore — 如果存在
✅ LICENSE — 如果存在

需要排除：
❌ .claude/settings.local.json — 包含个人权限配置
❌ .env / *.env — 环境变量/密码
❌ /tmp/* — 临时运行数据
❌ __pycache__/ — Python 缓存
❌ contact/ — 用户的业务数据
❌ .DS_Store — macOS 系统文件
❌ 任何包含 password/token/key/secret 的文件
```

**展示给用户确认**：
> "以下文件将被打包：
> [文件列表 + 大小]
>
> 以下文件已排除（包含个人信息或非必要）：
> [排除列表]
>
> 确认无误？"

### 第 P2 步：安全检查（强制，不可跳过）

自动扫描所有待打包文件，检查是否包含敏感信息：

检查模式：
- 邮箱密码 / App Password
- API Key（以 `SG.` / `re_` / `sk_` / `ghp_` 开头的字符串）
- 硬编码的邮箱地址（发件人/测试邮箱）
- 绝对路径（`/Users/xxx/...`、`/home/xxx/...`）
- IP 地址
- token / secret / credential 相关的赋值语句

如果发现敏感信息：
> "⚠️ 发现以下敏感信息，推送前需要处理：
> - `scripts/xxx.py` 第 42 行：包含硬编码邮箱
> - `xxx.md` 第 15 行：包含绝对路径
>
> 我来帮你清理？"

### 第 P3 步：生成/更新 README.md

如果项目没有 README.md，自动生成。如果有，检查是否需要更新。

README 应包含：
1. **项目名和一句话描述**
2. **功能概览**（Skill 列表和简介）
3. **安装方法**（一键安装 + 手动安装）
4. **使用方法**（触发关键词 + 示例对话）
5. **项目结构**（文件树）
6. **配置说明**（如果需要环境变量等）

### 第 P4 步：生成/更新 install.sh

安装脚本需要：
1. 自动发现并复制所有 `.claude/skills/*.md` 到目标项目
2. 复制所有 `scripts/` 目录到目标项目
3. 安装 Python 依赖（如果有 requirements.txt）
4. 清晰的安装成功提示 + 使用说明

### 第 P5 步：生成 .gitignore

确保排除：
- OS 文件（.DS_Store）
- Python 缓存（__pycache__/）
- 环境变量（.env）
- Claude 本地设置（.claude/settings.local.json）
- 临时数据和用户业务数据

### 第 P6 步：推送到 GitHub

**优先用 GitHub CLI（`gh`）**：

```bash
# 检查是否已安装和登录
gh auth status

# 创建仓库并推送
gh repo create <仓库名> --public --source=. --push
```

**备选：GitHub MCP**：
```
1. mcp__github__create_repository — 创建仓库
2. mcp__github__push_files — 推送文件
```

**备选：手动 git**：
```bash
git remote add origin https://github.com/<用户名>/<仓库名>.git
git push -u origin main
```

**推送前再次确认**：
> "即将推送到 GitHub：
> - 仓库名：`<仓库名>`
> - 可见性：`public`
> - 文件数：`X 个`
>
> 确认推送？"

### 第 P7 步：推送后验证

> "✅ 推送成功！
>
> 🔗 仓库地址：https://github.com/<用户名>/<仓库名>
>
> 别人安装方式：
> ```
> git clone https://github.com/<用户名>/<仓库名>.git
> cd <仓库名>
> ./install.sh /path/to/project
> ```"

## 使用场景

### 场景 1：对话后一键生成 + 发布（完整流程）
```
[用户和 Claude 完成了一个多轮任务]
用户："把刚才的能力打包成 skill 发布到 GitHub"
→ G1-G6（生成 Skill） → P1-P7（打包发布）
```

### 场景 2：只生成 Skill 不发布
```
[用户和 Claude 完成了一个多轮任务]
用户："把这个流程固化成一个 skill"
→ G1-G6（生成 Skill），不进入发布流程
```

### 场景 3：打包已有 Skill 发布
```
用户："把 .claude/skills/ 里的 skill 打包发到 GitHub"
→ 跳过生成，直接 P1-P7（打包发布）
```

### 场景 4：只做安全检查
```
用户："检查一下这个项目有没有敏感信息"
→ 只执行 P2
```

### 场景 5：只生成 README
```
用户："帮我写个 README"
→ 只执行 P3
```

## 注意事项

- **绝对不能推送包含密码、token、个人邮箱的文件**
- 推送前的安全检查是强制步骤，不能跳过
- 生成的 Skill 必须通用化——去除对话中的特定案例细节
- 如果对话中没有创建代码文件，Skill 可以是纯 `.md`（很多 skill 不需要配套代码）
- 生成 skill 时要参考项目中已有的 skill 文件作为格式范例
- 仓库默认设为 public（用户可以改为 private）
